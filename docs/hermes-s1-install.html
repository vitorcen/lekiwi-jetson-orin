  • API Server + gateway systemd user service 起在 127.0.0.1:8642，SSE 事件流完整可读 ✅；faster-whisper 中文逐字正确 ✅；edge-tts 直连微软成功 ✅。
  • 唯一缺口：环境继承了一个无效旧 DeepSeek key，待填有效 DEEPSEEK_API_KEY 后 TUI 对话 gate 才算过；nvpmodel -m 2 需 sudo，留给用户。

1. 硬件与音频Audio device baseline

项值
设备Lenovo MCP01 USB 会议麦克风音响（UAC 免驱，即插即用）
ALSA 卡名MCP01，寻址用 plughw:MCP01（按卡名，不用易变的 hw:N 序号）
放音48 kHz / 16 bit 立体声，走 PCM
录音48 kHz / 16 bit 单声道
mixerMic capture 100%，playback 走 PCM
❌ 坑 1：设备"插了没反应" — speaker was powered off 第一次插入内核只有一行 new full-speed USB device number 13 然后设备消失、无法枚举。原因不是驱动 —— 是音箱本体没开机。按开机键后立即枚举成功、ALSA 出卡。
❌ 坑 2：自放自录测试无效 — on-device AEC cancels the loopback MCP01 内置硬件回声消除（AEC），自己喇叭放的声音会被自己麦克风消掉，录到的只剩残渣，喂给 whisper 会输出典型幻觉文本（如"请点赞订阅转发"）。
正解：验证 ASR 用干净音频文件或真人说话，绝不能靠声学回环。
⚠ 坑 3：whisper 对近静音音频出幻觉 — hallucination on near-silence near-silence 输入下 whisper 会凭空吐字幕（如"字幕by索兰娅"）。生产链路必须 VAD 前置，只在检测到语音时才送 ASR。

3. Hermes 安装Install & pin

① uv + 官方安装脚本（换镜像）

# uv 0.11.29：板上无系统 pip3，先经 miniconda base pip 装，再 symlink
~/miniconda/bin/pip install uv==0.11.29
ln -sf ~/miniconda/bin/uv ~/.local/bin/uv

# 官方脚本下载后改仓库地址走 ghfast.top 代理
curl -fsSL https://hermes-agent.nousresearch.com/install.sh -o /tmp/hermes-install.sh
sed -i 's#REPO_URL_HTTPS=.*#REPO_URL_HTTPS=https://ghfast.top/https://github.com/NousResearch/hermes-agent.git#' \
    /tmp/hermes-install.sh

# 环境变量：PyPI 走清华 TUNA，python-build-standalone 走 ghfast.top 前缀
export UV_DEFAULT_INDEX=https://pypi.tuna.tsinghua.edu.cn/simple
export UV_PYTHON_INSTALL_MIRROR=https://ghfast.top/https://github.com/astral-sh/python-build-standalone/releases/download
bash /tmp/hermes-install.sh
项值
版本Hermes v0.18.2（build 2026.7.7.2），commit e598cef8
方式git 安装，位于 ~/.hermes/hermes-agent
Pythonuv 自带 py3.11，不动系统 3.10
依赖node v22 已有、ripgrep 14 已有；ffmpeg 8.0.1 用 conda-forge 装进 miniconda base（免 sudo）再 symlink ~/.local/bin
⚠ 版本已 pin — all config fields verified against v0.18.2 source 下文 profile 的每一个字段都对着 v0.18.2 源码核实过，非猜测。后续升级 Hermes 需重新核对字段名。
⚠ 坑 5：安装器尾巴报 "ripgrep not found" 是误报 — PATH timing false alarm 安装收尾时提示 ripgrep not found，实为 PATH 时序问题，rg 实际在（rg 14 已装）。忽略即可。

4. robot profile 配置DeepSeek routing & toolset trim

hermes profile create robot
# → ~/.hermes/profiles/robot/，wrapper 命令名 robot

config.yaml（字段全部经 v0.18.2 源码核实）：

model:
  provider: "custom:deepseek"     # custom: 前缀在 provider 字段
  default: "deepseek-v4-flash"     # bare 名，不带 custom: 前缀

providers:
  deepseek:
    api: https://api.deepseek.com
    key_env: DEEPSEEK_API_KEY
    transport: openai_chat

agent:
  disabled_toolsets: [terminal, file, browser, cronjob, skills_hub]  # kill-list 最后生效
  max_turns: 40
  api_max_retries: 3

platform_toolsets:
  cli: [web, vision, todo, skills]

skills:
  creation_nudge_interval: 0
⚠ toolset 粒度与裁剪边界 — read/write bundled, no standalone skill switch 裁剪是 toolset 粒度：file toolset 读写捆绑，不能只禁写；skill 自建无独立开关，只能靠禁 file + skills_hub 间接约束。disabled_toolsets 是 kill-list，最后生效、优先于 platform_toolsets 里的启用项。

.env（600 权限，占位值，永不进 git）：

DEEPSEEK_API_KEY=<key>              # 占位，只存板上
API_SERVER_ENABLED=1
API_SERVER_KEY=<openssl rand -hex 32 生成>   # 只存板上，不外泄
API_SERVER_PORT=8642
# gateway：装成 systemd user service + linger，监听 127.0.0.1:8642
robot gateway install
# → hermes-gateway-robot.service

5. 验证 gate 结果Acceptance gates

API Server（Bearer 鉴权 + SSE 流）

# 无 / 错 Bearer → 401
curl -s http://127.0.0.1:8642/api/sessions            # 401
curl -s -H "Authorization: Bearer $KEY" .../api/sessions   # sessions JSON

# 建会话 + 流式对话
curl -s -H "Authorization: Bearer $KEY" -X POST .../api/sessions \
     -d '{"id":"robot-main"}'
curl -N -H "Authorization: Bearer $KEY" -X POST \
     .../api/sessions/robot-main/chat/stream -d '{...}'
✓ SSE 事件序列完整可读 run.started → message.started → assistant.completed → run.completed → done，鉴权行为正确（无/错 Bearer 401、对 Bearer 出 JSON），POST 建会话 robot-main 成功。
⚠ 请求确实路由到了 DeepSeek，但 key 无效 — inherited a stale key 收到 DeepSeek 格式的 401（证明路由正确到 api.deepseek.com），但环境里继承了一个无效旧 key。待填入有效 DEEPSEEK_API_KEY 后，TUI 对话 gate 才算真正通过。
