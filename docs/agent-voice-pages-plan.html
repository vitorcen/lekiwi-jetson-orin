3.2 板端统一 config（新增，desired state）

位置 ~/.config/lekiwi/config.json（XDG 标准位：不在源码目录、天然不入 git、 可 grep 可手改）。入 git 的是 board/config.example.json—— 这是整个板端统一 config 的样例，不是 voice 专属：当前选了哪个 LLM （brain/preset）、全部 presets 与各自的 ASR/TTS 搭配、omni 地址、vision_speak 等开关， 全字段带注释充当 schema 文档，后续板端其它服务的配置也归并进来。之所以不放进 board/home/jetson/ 部署镜像树——那棵树会被 rsync 覆盖部署，而本文件是 daemon 运行时要写的（preset/开关），部署不能踩掉用户存的搭配。安装路径： voice/setup.sh 首次装机缺失才拷贝 example → ~/.config/lekiwi/config.json，已存在则不动。 voice-daemon 是唯一写入口（原子 write-then-rename）；其它板端服务 按需只读它（或由 daemon 下发），消费两式：

  • 直读：voice-daemon 自己的 ASR/TTS/brain/vision 播报开关。
  • 下发：Hermes 这种自带配置文件的，由 daemon 从统一 config 补丁其 config.yaml（见 §5.5）。下发结果即 applied 状态。
{
  "brain":  { "kind": "hermes", "preset": "deepseek" }, // hermes | omni(P4)
  "omni":   { "url": "http://192.168.3.2:8093" },
  "vision_speak": false,
  "presets": {
    "deepseek": { "api": "https://api.deepseek.com", "model": "deepseek-v4-flash",
               "transport": "openai_chat", "key_env": "DEEPSEEK_API_KEY",
               "pair": { "asr": "sensevoice",
                         "tts": { "engine": "edge", "voice": "zh-CN-XiaoxiaoNeural",
                                  "fallback": "melo" } } },
    "mimo":  { "api": "https://<OpenAI 兼容端点>", "model": "<MiMo 型号名>",
               "transport": "openai_chat", "key_env": "MIMO_API_KEY",
               "pair": { "asr": "sensevoice",
                         "tts": { "engine": "melo" } } }

}
状态模型：一份持久 + 一层临时，没有第三种 持久引擎状态只有 presets[*].pair + 当前 brain—— 运行引擎永远等于「当前 preset 的 pair」。Voice 调试页的引擎改动是内存态 ephemeral 覆盖：不落盘，退出调试态（或 daemon 重启）自动回到 pair。 Agent 页改搭配 = 直接改当前 preset 的 pair（落盘）。这消除了评审指出的 「当前值 vs 搭配值」双状态漂移。
  • /health 增报 {desired, applied, drift}：desired = config.json， applied = 实际加载的引擎 + 实际 yaml 中的 provider/model，drift = 两者差异 （如人手改了 yaml）。有 drift 只报不自动纠——纠偏是人的动作。
  • POST /config 语义为按轴整体替换（GUI 从 GET 拿全量、 改完整轴发回），不做深/浅合并猜谜。
  • key 永不进任何 config：preset 只存 key_env 名，值由人手工 写进板端 ~/.hermes/profiles/robot/.env（现有机制，P2 前置动作）。 daemon 切换前只校验该变量存在且非空，缺失 → 409 + 明确提示，GUI 永远看不到值。

5 · 板端改造：voice-daemon

5.0 前置实验（P0 第一件事，结果定架构）

两个实验，先做再写代码 ① 进程内卸载：加载 SenseVoice → del + gc.collect() → 读 /proc/self/status VmRSS 是否回落（onnxruntime arena + glibc 碎片大概率不还）。 不回落 → 引擎宿主定为子进程模式：每个 ASR/TTS 引擎一个 worker 子进程 （stdin/stdout 传 PCM/文本），切换 = kill 旧 + spawn 新，内存归还由 OS 保证。 ② 代理长请求：经 voice_post 打一个挂 20s 的端点，确认 15s 超时行为—— 这决定了所有长操作必须 202+job（已按此设计）。实验结果记回本文档。
实验① 结果（2026-07-21 实测，板 idle、voice-daemon 临时停） 用 daemon 同参数在 ~/voice/.venv 一次性脚本测进程 VmRSS（MB）：
引擎加载前加载+推理后del+gc.collect() 后+malloc_trim(0) 后
SenseVoice ASR29.0608.8303.3（仍 +274）47.1（+18.2）
Melo TTS47.1455.9301.5（仍 +254）48.3（+1.2）
结论：gc.collect() 单独不还 RSS（onnxruntime arena + glibc 碎片，正如预判）； 但 ctypes.CDLL("libc.so.6").malloc_trim(0) 有效——两引擎卸载+trim 后 RSS 均回落到加载前 +50MB 内（ASR +18.2MB、TTS +1.2MB，判定「归还」）。 → 引擎宿主定为进程内模式：unload() 序列必须以 malloc_trim(0) 收尾， 否则 RSS 不归还；子进程宿主非必需，省去 stdin/stdout 传 PCM 的复杂度。 （实验②「代理长请求 15s 超时」本轮未测，仍待做。）

5.1 引擎抽象

class OfflineAsr:    load()/unload()/transcribe(pcm) -> text     # sensevoice
class StreamingAsr:  load()/unload()/feed(pcm) -> partials, final # zipformer, P3 一等公民接口
class Tts:           load()/unload()/speak(text, generation)/cancel()  # 统一的是控制面
REGISTRY = {"asr": {...}, "tts": {"edge": EdgeTts(fallback=melo), "melo": ...}}
  • TTS 统一的只是 speak/cancel 控制面与事件——edge（异步网络流+ffmpeg+aplay+熔断） 与 melo（本地批量合成）实现自由，不硬凑 synth()->pcm。 edge→melo 熔断链保留在 edge 引擎内部，切换逻辑不感知降级态。
  • VAD/采集管线归 daemon 所有，不进引擎；流式引擎（P3）经 feed() 收连续帧， VAD 作为其实现细节内移。

5.2 切换执行器（唯一切换路径）

  • 全局 asyncio.Lock 串行化所有切换（/config、/brain、debug 覆盖）； 已有切换在跑 → 409 in-progress。
  • 切换序：置 switching 态（拒新轮次、interrupt 在途播放、generation++）→ drain _asr_pool/_tts_pool 在途任务（防推理中 unload 的 use-after-free segfault）→ 卸旧 → 载新（或 kill/spawn worker）→ 成功落盘。
  • 不承诺原子回滚：失败时尽力重载旧引擎，但旧引擎也可能因内存载不回—— 一律报真实 applied 状态（可能是 degraded: asr=none）， GUI 红字显示，由人处置。一次只切一条轴。

5.3 端点一览（Rust 代理白名单同步放行，列入阶段契约）

端点语义

5.5 大脑下发：Hermes yaml 补丁事务

  1. 前置校验：key_env 在 ~/.hermes/profiles/robot/.env 存在且非空； api 强制 https://（云端）且 scheme/地址过白名单校验（拒 link-local/metadata， 防 SSRF）；provider 名/model 名格式白名单。
  2. 持锁读 yaml → 记 hash → 备份 config.yaml.<hash前8>.bak → 只允许两类修改：改 model.provider/model.default 两键 + 幂等 upsert providers.<name> 段（存在则校验一致，不一致报 drift 拒改； 不存在才插入，含 transport）。其余键（尤其 disabled_toolsets、 mcp_servers）结构 diff 必须为空，否则中止——安全链不许被 换模型顺手动到。
  3. systemctl --user restart hermes-gateway-robot → 轮询 :8642 就绪 → ensure_hermes_session() 重建会话 → 探针轮：无工具、临时 session、1-token、≤10s 的真实 completion 打到新 provider。
  4. 探针过 → 落盘统一 config + 应用 pair；探针败/超时 → 按 hash 还原备份、重启网关、 再探针旧模型，全程 feed 回报。进程崩溃恢复：启动时发现 .bak 与 yaml hash 不一致 → 报 drift 待人处置，不自动选边。

5.6 Vision 播报桥（板端）

vision_speak=true 时 daemon 起后台任务：轮询 vlm-daemon caption 端点 （带 token，1-2s 间隔，成为订阅者使 vlm 保活），caption seq 去重 + 限长（>120 字截断）； 经内部 say 队列 latest-wins——正在播则丢弃旧帧不排队。SPEAKING（对话回答中）时暂停桥， 对话优先。

方向定谳、细节到 P4 再定稿（评审共识：这是另一套完整语音系统，不与 P0-P3 纠缠）：

7 · 内存（以实测口径为准，纸面数字不作数）

评审指出文档旧版 955MB 基线与实测 647MB RSS + 327MB swap（2026-07-20 记录）口径冲突。 P0 先建基线：冷启动/预热/稳态三点，采 PSS + private dirty + swap （RSS 受 paging 干扰不单独作数），记入本节。此前所有「切换省/费多少」一律标未知。 不变的硬规则：任何双 ASR/双 TTS 大模型并存禁止；引擎宿主已定为进程内模式 （§5.0 实验①），切换时内存归还由 unload()+malloc_trim(0) 保证（RSS 回落到加载前 +50MB 内）。

基线（2026-07-21 实测，voice-daemon，PSS+Private_Dirty+Swap，MB）。 load_models() 载入时同步预热（tts.generate("好")），故「刚就绪」即「预热后」：

数据点PSSPrivate_DirtySwapRSScommit≈PSS+Swap
冷启动/预热（刚就绪 ~90s）764747182773~946
稳态（就绪 2min，fresh）761743185770~946
稳态（长运行 aged，内存压力换出）305300646314~951

洞见：冷启动与长运行是同一 working set（commit ~950MB），只是 RSS/Swap 分布随运行时长与内存 压力而变——旧版「955MB 纸面 vs 647MB RSS+327 swap」的口径冲突实为同一足迹的不同分布，纸面 955MB 与实测 commit ~950MB 吻合。单引擎（ASR 或 TTS）常驻峰值见 §5.0 实验①：ASR ~580MB、TTS ~410MB 增量（加载+推理时的 RSS 涨幅）。
