RDK CS130WI/GS130WI（SC132GS）→ Jetson Orin CSI 驱动移植方案 Porting the D-Robotics GS130WI stereo camera (dual SC132GS + IMU) to Jetson Orin Nano native CSI

目标板：Orin Nano 8GB ｜ L4T R36.5.0 / 内核 5.15.185-tegra（JetPack 6.2.2）｜ 相机：RDK 双目模组（官方文档型号 GS130WI：双 SmartSens SC132GS 全局快门 1.3MP + ICM-42688-P IMU，一体模组，左右两条 22-pin FPC 各占一个 CSI 口。传感器实测为彩色 Bayer（BGGR）、无 IR-cut——RDK X5 上 pre-ISP RAW 相位统计定谳，见 §3）｜ 状态：方案设计，未实施 ｜ 日期：2026-07-18（依 codex 评审 + RDK X5 实测数据修订）

TL;DR

  • 拓扑先搞对：这不是「两个单目相机」，是一块双目一体模组——右排线走右目 + IMU（ICM-42688-P），左排线走左目。排线布 2 条数据 lane，但 X5 实机 1088×1280 寄存器表走 1-lane@1.2Gbps（30/60fps 同 PLL；896×896 才是 2-lane）——lane 数跟表走，别想当然按 2 配。模组只吃 3V3，没有 MCLK 输入脚（自带时钟），17 脚 RESET、18 脚 FSYNC。One stereo module, two FPC cables; self-clocked, no MCLK input; proven 1088×1280 tables are 1-lane.
  • 传感器是彩色不是单色：RDK X5 上抓 pre-ISP RAW 做 2×2 相位统计——绿对角相等（64.7/64.7）、蓝位低、红位高，BGGR Bayer 实锤。但模组无 IR-cut（NIR 850/940nm 增强是卖点），室内近红外污染 R/B，色彩仅供参考；深度/VIO 用亮度即可。Measured color BGGR, no IR-cut — treat color as best-effort.
  • 出厂标定白捡：每条排线上 0x50 EEPROM 存出厂双目标定（魔数 "UNION"：双目内参/畸变/R/T，基线 69.7mm）——Jetson 侧直接解析复用，免自标定（解析布局与代码见 RDK-experience 仓 stereo_cam.py parse_calib）。
  • 板上现状：无 /dev/video*；相机 I2C（i2c-2）当前 mux 分支扫到 0x50/0x58/0x68——0x68 与 ICM-42688-P 相符（旁证右排线在这个分支上）。sensor 地址待实测（0x30/0x32/0x33 都要试，见 P0）。
  • L4T 没有 SC132GS 驱动但不用从零写：地瓜开源 x5-libcam-sensor 出寄存器表，Rockchip sc132gs.c 出寄存器访问/控制参考（供电时序不适用——模组自供电），L4T nv_imx219.c 出 tegracam 框架骨架。
  • 出图路线：Bayer RAW10 直读（BG10）——sensor 本来就是 BGGR，DT 按 bayer/bggr 如实申报（不再是"伪装"），应用侧自行去马赛克，或 2×2 四像素合并得半分辨率亮度图（544×640，深度/VIO 够用）。Argus 理论上能吃 bayer 但没有这颗的 ISP tuning + 无 IR-cut 色彩没救，全程 V4L2 RAW 路线。
  • 工作量按里程碑分级：单路首帧 4–8 天；双路自由运行 + 控制齐全 + 稳定性验证累计 2–3 周；硬同步 + IMU 时间戳 + VIO 可用累计 3–6 周。前置死关：22-pin 电气映射逐 net 核对——过不了直接放弃，换 IMX219。

3. 管线原理：RDK X5 vs JetsonPipeline anatomy — why the userspace lib can't be "ported" as-is

地瓜的「驱动」活在自家 VIN/ISP 用户态框架里；Jetson 的相机活在内核 V4L2 + tegracam 里。两边从 MIPI 往上没有一层是对得上的——所以移植的本质是用地瓜的数据重写一个 Jetson 驱动，不是搬代码。

⚠ CFA 事实与出图路线 — the sensor is color BGGR; routes 早期按「单色」规划——已被实测推翻：RDK X5 上抓 pre-ISP RAW（同款模组、同 1088×1280 表）做 2×2 相位统计，绿对角相等（64.7/64.7）、蓝位 48.8、红位 60.7，BGGR Bayer 彩色实锤。但模组无 IR-cut（NIR 增强设计），室内近红外污染 R/B——色彩只能当参考，深度/VIO 用亮度。注意：不走 Argus ≠ 不走 tegracam——内核侧仍是 tegracam sensor 框架，只是用户侧走 V4L2 RAW。
路线 A（首选）：Bayer RAW10 如实申报——DT mode_type="bayer"、pixel_phase="bggr"，驱动报 MEDIA_BUS_FMT_SBGGR10_1X10（V4L2 fourcc BG10）。应用侧两个消费法：① 去马赛克得彩色（NIR 污染，仅监控用）；② 2×2 四像素平均得 544×640 亮度图（CFA 各相位等权，天然抵消——深度/VIO 的正解）。bytesperline/16-bit 容器对齐要实测。上板后用同一套 2×2 相位统计自检（绿对角应相等）确认相位没被 flip/crop 搬动。改动全部圈在自己的驱动+DT 里。
路线 B（二期，风险高）：原生 Y10——r36.x 的 DT 属性字典只有 bayer/yuv/wdr，要动 sensor_common/camera_common/vi5_formats 并重编 NVIDIA camera core OOT 模块；且 sensor 实为 Bayer，Y10 还得先在 sensor 内合并——价值存疑，基本可划掉。

4. 实施计划Implementation plan — phased, each phase has a kill switch

每个阶段有明确的通过判据，不过就地止损——尤其 P0，电气不兼容就直接放弃，一行驱动都别写。

P0 · 电气兼容性验证（1 天）—— 先决条件，过不了全盘放弃

已知两端完整引脚（模组来自 官方硬件页，Jetson 来自载板规格书 Table 3-1）。模组 Pin1=GND … Pin22=3V3，Jetson J20/J21（Molex 54548-2272，下接触、金属面朝散热器）Pin1=3.3V … Pin22=GND——两端互为镜像。逐 pin 按反序（模组 m ↔ Jetson 23−m）核对，22 个里 21 个严丝合缝，唯一例外是 pin 18：

⚠ pin 18：不是短路，是软件层要关掉的一根线 — the one exception, handled in DT not hardware 反序后 Jetson 的 CAM0_MCLK（输出）物理接到模组 FSYNC（输入）。两端都是 1.8V 信号线，不是电源，插反也不烧；问题只是若 DT 使能 MCLK，Jetson 会往这根线打 24MHz 方波灌进模组 FSYNC。对策：DT 里不配 mclk（这颗模组自带时钟不需要），该脚留 GPIO/不驱动即可；且自由跑表（0x3222=0，X5 实证）本就忽略 FSYNC——功能上无害。反过来看这还是个便利：将来要硬同步，正好可把这根 GPIO 主动拉去驱动 FSYNC，见 P4。

两个结论：① 模组只用一组 2-lane，落在连接器 CSI1 brick；模组 pin 11/12/14/15 是 NC 空脚，反序后正好盖在 Jetson 的 CSI0 数据线上——不是冲突，是 Jetson CSI0 整组空用（P2 设备树只配 CSI1）。② 这套镜像反序正是树莓派 22-pin 相机约定，Orin Nano 本兼容树莓派相机，模组随附线（本给同约定的 RDK X5）大概率直接可用。

✗ 但"插异向排线就行"仍是待实测，不是已验证 — still to be verified, not confirmed 「镜像对得上」是纸面推演；实际该用同面还是异面 FPC，取决于两端连接器接触面朝向，光看引脚表定不了。而插反的失败模式是破坏性的（模组 3V3 撞 Jetson GND）。所以上电前必须万用表验反序连通：断电测 Jetson pin1 ↔ 模组 pin22 通（=反序，对，可限流上电）；若测出 Jetson pin1 ↔ 模组 pin1 通（=直通线）→ 停，别上电，换线或翻面。

P4 · 双目硬同步 + IMU（独立子项目,3–10 天,按需）

  • 硬同步不是「DT 加个属性」：地瓜的 master 表实为自由运行、slave 表是外触发且标 30fps——60fps 对称同步未经验证。需要：外部 FSYNC 源（PWM/GPIO,经转接把信号送到两路 18 脚,核对 1.8V/3.3V 阈值）、驱动显式实现 free-run/slave 模式切换、示波器或闪光 LED 实测左右曝光相位差（不能只看 V4L2 时间戳）、处理双节点启动顺序与序号对齐。
  • IMU（ICM-42688-P @0x68）：r36.5 rootfs 没有现成 inv_icm42600 模块——要么使能/移植该 IIO 驱动,要么用户态 i2cdev 读 FIFO。「能读数」半天;VIO 可用（ODR/量程/FIFO/中断/时间戳/轴向/相机-IMU 外参标定/ROS 2 发布）是另外一周起步,单独立项。
