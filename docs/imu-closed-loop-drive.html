IMU 闭环运动控制:让 LLM 控车距离/方向更准

LeKiwi 三轮全向底盘 · Jetson Orin Nano 8GB · 2026-07-22 · Claude + codex gpt-5.6-sol 双模型头脑风暴定稿 · 关键代码断言已逐条核实

TL;DR

  • 数据结构错了才是根因:LLM 该提交「目标」(转 90°/走 1 米),不该自己拼「速度 × 时间」。开环 drive_move(v,ω,t) 的距离误差来自加减速斜坡、打滑、地面差异,LLM 端无法修。
  • 闭环放独立 ROS2 controller,不塞进 MCP、不污染 base_host。MCP 变薄适配器只交目标;base_host 只补齐它本来就欠的:sync-write、latest-only、applied ack、最终钳位。
  • 纯 IMU(无轮 odom)能闭环的只有 yaw:相对转向 ±2–3° 可期;平移距离闭不了,加速度二次积分是坑(0.01g 偏置 10 秒漂 4.9 米),不做。
  • 先修地基(P0):实测核实 base_host 每速度帧逐轮 txrx ≈72ms > 20Hz 周期 50ms,且队列逐帧执行陈旧命令——不修这两条,任何闭环都是在对着被延迟/丢弃的命令加力。
  • 路线:P0 修命令链 → P1 turn_by yaw 闭环 → P1.5 直行航向保持 → P2 轮 odom(sync-read ≥30Hz) → P3 move_by(dx,dy) → P4 受限 sequence。磁力计标定 MVP 阶段不做。

1 · 现状与问题

LLM 控车现走 drive/mcp_server.py 的 drive_move(vx,vy,omega,duration):开环速度 × 时间, 硬钳位 |v|≤0.15m/s、|ω|≤30°/s、≤2s,20Hz 重发 + 刹车帧 + 冷却 + 互斥。imu_read(2026-07-22 上线) 已能读姿态/航向/气压,但只是快照观测,不参与控制。

误差来源影响LLM 端可修吗
加减速斜坡吃距离0.15m/s × 2s 名义 0.3m,实际短 10–20%否——斜坡形状 LLM 不可见
全向轮平移时偏航「直行」走成斜线/香蕉线否——需要实时 yaw 反馈
ω × t 开环旋转「转 90°」±10° 起步,地毯更差否
地面差异(硬地/地毯)同一指令不同距离理论可标定系数,实践 LLM 会算错
多步任务误差累积三步之后位置已不可信否——一步 5% 三步 15%

结论:让语言模型兼职底盘控制器是数据结构层面的错误。该消灭的是 「LLM → 速度帧 → 时间到停」这条开环链,换成「LLM 交目标 → 控制器带反馈执行 → 返回实测结果」。

2 · 地基先修:base_host 命令链(P0,闭环前置条件)

codex 审查点出、并经本仓库代码逐条核实的四个问题——不修它们,闭环控制器会对着 被延迟或被丢弃的命令持续加力:

#问题(已核实)位置后果
1每个速度帧对 3 个轮子逐个 txrx:每轮 4ms sleep + read(64)(应答只有 ~6 字节,pyserial 等满 20ms 超时)≈ 24ms,3 轮 ≈ 72ms > 20Hz 周期 50msbase_host.py txrx() / drive()队列必然积压,停车帧排在旧速度帧后面
2poll 排空队列时逐帧执行旧命令,不是每源只留最新帧主循环 recv 段陈旧帧继续驱动轮子并喂 watchdog
3MCP 返回的 frames_sent 只代表进了 ZMQ 发送缓冲,不代表经仲裁写到了轮子drive/mcp_server.py闭环控制器不知道自己的命令是否生效(pad/GUI HOLD 窗口内 MCP 帧被静默丢弃)
4vx、vy 分量各自钳位 0.15:对角平移模长实际 0.212m/sdrive/mcp_server.py若安全定义是 |v|≤0.15,应钳向量范数

P0 改法(与 ROS2 集成计划已定稿的协议升级同一件事,不是新增工程):

  • 三轮速度一次 broadcast sync-write(STS 协议原生支持,无逐舵机应答);位置一次 sync-read。串口 1Mbps 带宽从来不是瓶颈,事务模型才是。
  • ZMQ 命令带 seq + 单调 ts + TTL;每源 latest-only,陈旧帧直接丢。
  • base_host 对外发布 owner / applied_seq / motion_on / ts——控制器由此知道「我的命令是否被采用」。
  • 最终速度/加速度钳位下沉到 base_host 仲裁之后,对 pad/gui/ros/mcp 全源生效;平移按向量模长钳。

3 · 闭环放哪层

方案优点致命伤判断
(a) MCP 进程内 20Hz 控制线程改动最小,可快速验证控制算法IMU 反馈绕 ROS2→rosbridge→WebSocket;生命周期与 LLM 会话耦合;不知道命令是否被仲裁采用;人接管时可能继续追误差只配当一次性实验,不是产品架构
(b) base_host 内做闭环独占串口可直读轮计数,离执行器最近把高层控制、串口、机械臂、仲裁、安全全塞一个进程;控制器 bug 可拖死 watchdog不做
(c) 独立 ROS2 motion_controller(推荐)直接订阅 /imu/data 零 rosbridge 延迟;天然支持 goal/cancel/result;进程崩溃由 base_host 0.5s watchdog 兜底停车;后续接轮 odom/EKF/Nav2 顺路多一个节点 + 必须先有 P0 的 applied ack✅
LLM
 → MCP(薄适配器):提交目标 turn_by / move_by
 → ROS2 motion_controller:订阅 /imu/data(后续 /odom),P 控制 + 终止判据
 → ZMQ 速度帧 (src=mcp, seq, ttl)
 → base_host:仲裁 → 最终钳位 → 安全门 → sync-write 三轮
 → 状态回流:owner + applied_seq + 轮反馈

闭环链路年龄预算(P0 修完后):IMU 25Hz 采样 0–40ms + ROS2 传输数 ms + 控制周期 40–50ms + ZMQ/仲裁/sync-write 数 ms → 常态 <80ms,p95 <120ms。rosbridge 只许传「目标/结果」, 不许进 20Hz 反馈主链。

4 · 给 LLM 的工具面设计

MVP 三件套

turn_by(angle_deg)     # >0 左转(逆时针), <0 右转;以当前姿态为零点,不依赖上电朝向
motion_stop()          # 取消当前自动运动,幂等;不宣称能压过人工控制
motion_status()        # goal 状态 / 实测已转角度 / owner / 反馈新鲜度
  • 速度、加速度、容差由服务器固定,不给 LLM 调——参数越少越难用错。
  • 保留 drive_move 兼容现有调用者(Never break userspace),但工具描述降格为「低层调试接口」,引导 LLM 用目标式工具。它同时是 IMU 失效时的显式降级路径:开环速度×时间不依赖 IMU/rosbridge/ROS,IMU 挂了照样能走;pad/GUI 遥控直连 base_host,更是全程不受影响。
  • motion_status 必须带 imu_ok(反馈新鲜度):闭环工具因 sensor_stale 拒绝后,LLM 据此确认 IMU 不可用,再明知故用 drive_move 开环执行并向用户声明「这步是估的」——降级永远由 LLM 显式选择并如实播报,禁止的只是闭环工具内部偷偷退化成按时间跑还谎称成功。
  • 结果返回「目标值 + 实测变化 + 最终误差 + 耗时 + 执行 owner」——把「ZMQ 发送成功」叫「执行成功」的时代到此为止。

有轮 odom 之后

move_by(dx_m, dy_m)    # body frame(该步开始时):dx>0 前进, dy>0 左移;钳 hypot(dx,dy)
turn_to(yaw_deg)       # REP-103 yaw(逆时针+),自动最短路径;仅同一 IMU epoch 内有效
execute_motion(steps)  # 受限序列,见 §8
接口符号地雷 imu_read 返回的 heading_deg 是顺时针罗盘制,底盘 theta 是逆时针 REP-103—— 两套符号同时喂 LLM 必出符号事故。目标式工具统一只用 REP-103 yaw;turn_to 不叫 heading。IMU 重启(epoch 变)后旧的绝对目标必须拒绝。

busy / 抢断语义

  • 全局唯一自动运动 goal,不排队;busy 时返回 busy + active_goal_id。
  • pad/GUI 接管 → 控制器立即以 preempted_by_human 结束,绝不在 HOLD 窗口过后偷偷续跑;人工接管后必须重新提交目标。
  • 状态枚举不能只有成功/失败:succeeded / busy / canceled / preempted / blocked / sensor_stale / no_progress / timeout / controller_fault——每个都是 LLM 能理解并据此重规划的信号。

5 · 无轮 odom 时,纯 IMU 的能与不能

✅ 能闭环❌ 闭不了
相对 yaw 转向(turn_by)平移距离(只能 v×t 估计,结果须标记 distance_estimated=true)
平移时航向保持(直行不跑歪——单项收益最大)横向漂移、打滑后的真实位置
倾斜/碰撞/静止检测基于未标定磁场的绝对朝北
校验轮 odom 的旋转方向与漂移(P2 起互检)加速度二次积分当里程计
加速度积分,死心 0.01g 恒偏置 → 10 秒位置误差 0.5×0.098×10² ≈ 4.9 米;1° 倾角误差把重力投影成 0.17m/s² 假加速度;再叠轮振 + 25Hz 采样。它只配做短时「动没动」检测。

yaw 闭环控制律(P 控制,不要积分项)

error = shortest_angle(target - current_yaw)
omega_cmd = clamp(kp * error)          # + 角加速度限幅 + 近目标减速
# 成功判据: |error| < 2° 且 |gyro_z| 足够小,连续 3–5 帧
# 0.5s 无有效进展 → no_progress 停车
# 积分项在静摩擦/被人工压制/轮卡住时只会积累灾难——不加

精度预期(先按此对外承诺,实测后再收紧):30°/s 时 25Hz 每帧转 1.2°,末段降到 8–10°/s 每帧 0.3–0.4°;带减速段 + 稳定窗口,硬地重复误差 ±2–3° 合理,对外按 ±5° 讲。 ±30/±90/±180° 各测 5–10 次、硬地和地毯分开,达不到就看数据,不拍脑袋调 PID。

6 · 轮 odom(P2):直接上 sync-read 30Hz,不做低频逐轮读

  • 0.15m/s 下 5Hz 反馈每周期已走 3cm;30Hz 只 5mm——终点减速与异常检测质变。
  • 一次 sync-read 三轮 2 字节位置,指令+应答几十字节,1Mbps 下线速 <1ms。贵的是现在的固定 sleep + 超时读 + 逐舵机事务,不是带宽。
  • 4096 counts/圈、轮周长 ~0.314m,分辨率远非瓶颈;打滑和采样延迟才是。
  • 软件展开多圈计数(STS3215 掉电不存),检测跳变/重启/采样空洞;先分别验证轮 odom 与 IMU,再谈 EKF——EKF 不是掩盖坏标定的化妆品。
  • 机械臂命令降频、合并,轮控制周期优先调度(同一条 1Mbps 串口还挂着 SO-101 六舵机)。

8 · 安全:闭环的新风险

开环错误跑一段就停;闭环错误会永远坚持错误方向——控制器持续纠错是新增风险的根。必须加:

  • 单目标上限:最大角度/距离/执行时间;超时即 timeout 停车。
  • IMU 消息年龄 >120–200ms → 立停当前自动目标;quaternion 无效、yaw 突跳、IMU epoch 变 → 立停。不允许「IMU 坏了就退化按时间跑」的 fallback。
  • applied_seq 超时或 owner 不是本控制器 → 以 preempted 结束。
  • 有命令但 yaw/odom 无进展 → blocked;误差反复过零/振荡过多 → 停车,不是调参硬顶。
  • 控制器进程退出 → base_host 0.5s watchdog 兜底(第 3 层不变)。

雷达净空 gate

要进,但位置守住原计划:base_host 仲裁之后、带 TTL 的裁决、覆盖所有速度源(不只挡 ROS)。 分两步:yaw MVP 有人监督时可先不做发布门槛(限低速 + 清空旋转包络);任何超过现有 0.3m 单次风险包络的 move_by / sequence / 无人值守任务,必须先有 gate。gate 按速度方向算扫掠区 (前/后/横移分向检查;旋转用整车外接圆,臂伸出时扩 footprint;含制动距离+余量;/scan 超时即停), 并明说 2D 雷达看不见桌面、悬空物、透明物和悬崖——别把它包装成完整避障。

9 · 组合任务:先多次调用,后受限 sequence

MVP 阶段让 LLM 多次调用:每步可观察、可停止、易定位错误,错误恢复语义天然 (读 motion_status → 按实际位置重新规划)。轮 odom 与错误语义稳定后,再提供原子 execute_motion:

{"steps": [
  {"op": "move_by", "dx_m": 1.0, "dy_m": 0.0},
  {"op": "turn_by", "angle_deg": -90},
  {"op": "move_by", "dx_m": 0.5, "dy_m": 0.0}
]}
  • 只允许 move_by / turn_by / wait——不支持脚本、循环、条件。
  • 执行前一次性验证每步与总距离/总角度/总时间;≤8 步。
  • 全序列持同一运动 lease;任一步失败立即停车,后续不执行;返回 completed_steps / failed_step / reason / measured_pose_delta。
  • 绝不从失败点自动恢复——LLM 先读状态再生成新计划。
自然语言歧义不归 sequence 管 「前进 1 米右转 90 度回来」——「回来」是倒退 1 米还是回到原点?sequence 只执行明确计划; 涉及返回原点,LLM 应先复述路径或要求确认(SOUL.md 运动守则已有「含糊先确认」)。 waypoint 现在不做:没有可靠 /odom,waypoint 只是换名字的开环时间控制。

10 · 分阶段路线

阶段内容验收门
P0 地基三轮 sync-write;每源 latest-only + seq/ts/TTL;发布 owner/applied_seq;最终钳位(含向量模长)下沉 base_host;实测停车延迟20Hz 速度帧无积压;停车帧不再排队;MCP 能确认命令被采用
P1 yaw 闭环 MVP独立 ROS2 motion_controller 订阅 /imu/data;MCP 加 turn_by/motion_status,drive_stop 可取消 goal;保留 drive_move±30/±90/±180° 硬地重复误差 ≤3°,最坏 ≤5°
P1.5 航向保持平移仍按时间估距,IMU 动态补偿 ω 防跑歪;结果标 distance_estimated=true1m 直行横向偏差显著小于开环
P2 轮 odom三轮位置 sync-read ≥30Hz;软件展开计数;ROS 节点算正运动学;硬地/地毯 vx/vy/ω 标定轮反馈 ≥30Hz、p95<40ms(与 ROS2 计划 P2 同门)
P3 move_by轮 odom 控距离 + IMU 保 yaw;减速/终点稳定/打滑异常;雷达 gate 前保持 0.3m 单次包络1m 平移误差 ≤3–5cm(实测定)
P4 组合任务受限 execute_motion;/odom 稳定后再谈相对 waypoint;有定位后才有 navigate_to(map_pose)失败即停、无自动续跑经混沌测试

附 · 双模型结论与分歧

  • 一致:先闭环 yaw;LLM 交目标不拼速度帧;磁力计 MVP 不标;加速度积分不做;组合任务先多次调用。
  • codex 修正了 Claude 的初始倾向:Claude 起初倾向「MCP 进程内 20Hz 控制线程」求最小改动;codex 指出反馈绕 rosbridge、生命周期耦合、以及「控制器不知命令是否被仲裁采用」三点,并给出 base_host 串口事务实测级问题(逐轮 txrx ≈72ms)——P0 是前置条件,不是顺手优化。Claude 逐条核实代码后采纳:小型 ROS2 yaw controller + 薄 MCP 目标适配器。
  • 与既有 ROS2 集成计划零冲突:P0 的 seq/ts/TTL/latest-only 与计划中的命令协议升级是同一件事;P2 轮 odom 与计划 P2(:5557)同门;雷达 gate 位置(仲裁后、全源生效)与计划的 lekiwi_safety 裁决一致。
