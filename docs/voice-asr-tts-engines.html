语音引擎全景：ASR 多宿主 + TTS 三引擎

Jetson Orin Nano 8GB · voice-daemon (:8092) · 2026-07-22 · 全部数据为板上实测

TL;DR

  • ASR 门面引擎 = Fun-ASR-Nano 0.8B(GPU, ~0.65s/句,带标点)；抗噪备选 qwen3；流式仅调试用。
  • TTS：edge(在线主力)→ matcha(离线实时,RTF 0.18)→ melo(兜底常驻)。melo RTF 1.60 不实时且丢数字——它当主引擎的时代结束了。
  • 持久状态只有一份：presets[*].pair；Voice 页改动是临时覆盖，「保存」扇出写入所有 preset——Agent 页切大脑不会顶回旧引擎。
  • 质量上限路线（待实验）：Qwen3-TTS-0.6B 走 TensorRT-Edge-LLM(int4, ~1.7GB, 原生流式)。

1 · 架构:一份持久状态 + 数据驱动多宿主

运行引擎永远是「当前 preset 的 pair」的投影。Voice 调试页的引擎/参数改动是内存态 ephemeral 覆盖(不落盘,退出调试自动还原);「保存」把值写回 pair 并扇出到所有 preset——per-preset 分歧没有任何 UI 能产生它,保留它只会让 Agent 页切大脑时把 ASR/TTS 悄悄顶回旧搭配(实锅已修,2026-07-22)。

axes:  brain(preset) | asr | tts | vad | audio | stream(DEBUG-only) | vision
hosts: REGISTRY["asr"] = funasr qwen3 sensevoice paraformer whisper
       REGISTRY["tts"] = melo matcha       # edge 是 melo 之上的播放模式
切换:  载新→卸旧;大宿主(funasr)unload_first 先卸旧防击穿 8GB
保存:  切运行引擎 + 落盘(扇出全 preset) + 清 ephemeral 覆盖 → 对话链即刻生效

2.1 流式模式(DEBUG-only,不进对话链路)

免 VAD、自带端点(OnlineRecognizer)。默认 x-asr-zh-en(160M/百万小时,中英混,RTF 0.25); 备选 zh-xlarge(700M,纯中文,RTF 0.56)、para-zh-en(int8 226MB,RTF 0.115)等。 三个已修的坑:①解码下专用线程(内联解码卡事件循环→ALSA overrun 整段丢音); ②端点检查移进 decode 循环(批中间的端点原来会整句丢);③num_threads=2 最优(4 线程是负优化)。 结论:这套硬件上流式是对比工具,不是可用模式——精度天生低于离线,丢块敏感,对话恒走 VAD+离线。

GPU 之于 ASR pip 版 sherpa 是 CPU-only;官方 cuda wheel 旧结论"绑 CUDA 10.2 不适配 JetPack 6"已过时—— sherpa-onnx 现已支持 Orin Nano JetPack 6.2 自编(ORT 1.18.1 + CUDA 12.6,板上正是 12.6)。 但离线 RTF 已够用,重开 ASR GPU 只在与 TTS GPU 自编合并时顺带考虑。

3.2 质量上限路线(待实验):Qwen3-TTS-0.6B

官方只有 torch/vLLM(fp16 2.5GB,无 ONNX/GGUF 上游)。Jetson 现实路径 = NVIDIA TensorRT-Edge-LLM(官方支持,拆 Talker/CodePredictor/Code2Wav 三引擎; 社区现成 int4+fp8 包 ~1.7GB,9 音色 zh+en,原生流式;Edge-LLM issue #87 有 Orin Nano 8GB 同板跑 ASR+TTS 先例)。坑:Predictor 每帧串行 15 步是天然瓶颈;llama.cpp sm_87 要关 flash-attn;官方 97ms 首包是云端数,板上预期 ×3-5;与 VLM 3.4GB 共存需先停 VLM 实测峰值。

GPU 收益评估结论 matcha CPU 已 8×实时 → 小模型上 GPU 不解决真问题;GPU 的用武之地只有两个: ① kokoro 音质路线(需自编 sherpa GPU,预期 5-10×),② Qwen3-TTS 质量线(必须 GPU)。 按需求触发,不为加速而加速。
