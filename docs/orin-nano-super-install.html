Jetson Orin Nano Super 刷机安装记录 Flashing Jetson Orin Nano 8GB to JetPack 6.2.2 + NVMe rootfs

目标板：Orin Nano 8GB devkit ｜ 系统盘：256GB M.2 NVMe ｜ 主机：Ubuntu 24.04 x86_64 ｜ 日期：2026-07-16

TL;DR

  • 刷 Jetson Linux 36.5（= JetPack 6.2.2，Ubuntu 22.04 / 内核 5.15），不是最新的 JetPack 7.2 —— 生态成熟度优先。
  • board target 用 jetson-orin-nano-devkit-super，走 Super 的 DTB/BPMP。
  • 一条 l4t_initrd_flash.sh 同时刷 QSPI bootloader + NVMe 完整系统。
  • 主机必装 nfs-kernel-server，否则直接 exit 110 —— 官方依赖列表里没写。
  • 67 TOPS 的 Super 模式靠开机后 nvpmodel -m 2 打开，不是靠 -super-maxn 配置。

1. 为什么是 JetPack 6.2.2 而不是 7.2Why 6.2.2, not the newest 7.2

3. 完整步骤Step by step

3.0 前置检查

板子进 recovery 模式（按住 FC REC 键再上电），主机确认：

lsusb | grep -i nvidia
# → Bus 001 Device 0xx: ID 0955:7523 NVIDIA Corp. APX   ← 必须看到这个
⚠ Type-C 线材是最常见的失败源 — cable quality matters initrd flash 全程跑 USB + NFS，掉一次线就得从头再来。必须用能跑数据的线，不是充电线。

3.1 主机依赖

sudo apt-get install -y qemu-user-static lbzip2 abootimg \
     device-tree-compiler nfs-kernel-server
❌ 坑 1：nfs-kernel-server 缺失直接 exit 110 — the undocumented dependency ERROR nfs-kernel-server not found! —— 官方文档的依赖列表里没提它，但 --network usb0 模式要靠 NFS 把 rootfs 喂给板子上的 initrd。装上即可。

3.2 下载并展开

L4T=https://developer.nvidia.com/downloads/embedded/L4T/r36_Release_v5.0/release
curl -L -O $L4T/Jetson_Linux_R36.5.0_aarch64.tbz2                        #  750 MB
curl -L -O $L4T/Tegra_Linux_Sample-Root-Filesystem_R36.5.0_aarch64.tbz2  # 1.8 GB

tar xf Jetson_Linux_R36.5.0_aarch64.tbz2                    # 普通用户即可
cd Linux_for_Tegra
sudo tar -I lbzip2 -xpf ../Tegra_Linux_Sample-Root-Filesystem_R36.5.0_aarch64.tbz2 -C rootfs/
sudo ./apply_binaries.sh

rootfs 必须用 sudo + -p 解压：要保留 root 属主与 setuid 位。普通用户解出来的 rootfs 刷进去是坏的。校验：sudo stat -c '%U:%G %a' rootfs/usr/bin/sudo 应为 root:root 4755。

3.3 刷机

sudo ./tools/kernel_flash/l4t_initrd_flash.sh \
  --external-device nvme0n1p1 \
  -c tools/kernel_flash/flash_l4t_t234_nvme.xml \
  -p "-c bootloader/generic/cfg/flash_t234_qspi.xml" \
  --showlogs --network usb0 \
  jetson-orin-nano-devkit-super internal
参数作用
-p "-c ...qspi.xml"刷 QSPI：bootloader / UEFI
-c ...flash_l4t_t234_nvme.xmlNVMe 分区布局。该 XML 里只有一个 <device type="external">，但包含完整启动链（esp / A_kernel / kernel-dtb / APP rootfs）
internal（rootdev）让 NVMe 用自己的 PARTUUID 作 rootfs。Orin Nano devkit 无 eMMC，XML 里也没有 internal 存储设备定义，所以"internal"那半不会真写
--network usb0经 USB 网络 + NFS 传输 rootfs

4. 踩过的坑Gotchas hit in practice

❌ 坑 2：-super-maxn 不适用于 8GB Nano — wrong config for this SKU jetson-orin-nano-devkit-super-maxn*.conf 只覆盖模块 SKU 0000/0002/0001（即 Orin NX 系列），Nano 8GB 是 3767-0003 / 0005，跑进去直接 exit 1。而且它改的只是 BPMP 超频 DTB，并非官方 Super 模式。
正解：board 用 jetson-orin-nano-devkit-super，67 TOPS 靠系统内 nvpmodel 开。
模块 SKU型号
3767-0000Orin NX 16GB
3767-0001Orin NX 8GB
3767-0004Orin Nano 4GB
3767-0003 / 0005Orin Nano 8GB ← 本机
⚠ 坑 3：Please install the Secureboot package ... for fused board 这条只是提示，不阻断流程 —— 非 secureboot 熔丝板可忽略，脚本会继续往下跑。
⚠ 坑 4：shell 退出码陷阱 — exit code swallowed by echo cmd; echo "done: $?" 这种写法，脚本整体退出码是最后那个 echo 的 0，会把失败伪装成成功。要判断刷机结果必须读日志，或显式 rc=$?; ...; exit $rc。
❌ 坑 5：刷机期间主机卡死重启 — host froze mid-flash 实测发生过一次：主机在刷机中途死机重启，进程连同 /tmp 下的日志一起消失。
对策：用 setsid nohup 让刷机脱离终端会话，日志写到 非 /tmp 的持久路径： 
setsid nohup sudo -A ./tools/kernel_flash/l4t_initrd_flash.sh ... \
  > ~/flash.log 2>&1 < /dev/null &
判断是否真的刷完：tools/kernel_flash/initrdlog/ 目录存在才说明进入了实际刷写阶段；images/ 为空 + 无 initrdlog/ = 根本没碰到 NVMe，可安全重刷。

5. 刷完之后After the flash

刷机结束板子自动重启，接显示器 + 键鼠完成 oem-config（建账号、语言、时区）。进系统后开启 Super 模式：

# 查看当前功耗模式
sudo nvpmodel -q

# 切到 MAXN_SUPER（67 TOPS），会提示重启
sudo nvpmodel -m 2
sudo reboot

# 验证
sudo nvpmodel -q          # → NV Power Mode: MAXN_SUPER
cat /etc/nv_tegra_release # → R36 REVISION: 5.0
sudo jtop                 # 需要先 pip install jetson-stats
✓ 成功判据 — how to know it worked ① 板子从 NVMe 启动，不再需要 SD 卡；② lsblk 显示 nvme0n1p1 挂在 /；③ nvpmodel -q 能切到 MAXN_SUPER；④ nv_tegra_release 显示 R36 REVISION 5.0。
⚠ 无线网卡要单独启用 — Wi-Fi needs a separate step Intel AX210 Wi-Fi 6E 的固件随系统预装，但 iwlwifi 驱动模块 L4T 默认没编，刷完机设置里看不到 Wireless。启用步骤（装 backport-iwlwifi DKMS + 一个 cfg80211 符号冲突的坑）见 Orin · AX210 Wi-Fi 6E 驱动启用记录。
